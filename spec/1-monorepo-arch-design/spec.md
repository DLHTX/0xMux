# 功能规格说明：0xMux Monorepo 架构设计与分发体系

**版本**: 1.0
**日期**: 2026-02-09
**状态**: 草案
**分支**: `1-monorepo-arch-design`

---

## 概述

0xMux 是一款面向开发者的 tmux Web 管理面板。本规格说明定义了项目的整体架构设计，涵盖：monorepo 代码组织、前后端通信机制、统一开发命令、打包构建流程、多渠道分发方式（npm/brew）、环境依赖检测、以及极客命令行风格的 UI 设计规范。

最终产品形态：用户通过 `npx 0xmux`、`npm i -g 0xmux`、`brew install 0xmux` 安装后，在本机运行一个服务（默认端口 1234），浏览器打开即可使用。

---

## 目标用户

- **主要用户**: 使用 tmux + Claude Code 的开发者
- **次要用户**: 任何需要通过浏览器管理 tmux 会话的终端用户

---

## 用户场景

### 场景 1：首次安装与启动（依赖完备）

**角色**: 开发者
**前置条件**: 用户本机已安装 Node.js 或 Homebrew，tmux 已安装
**流程**:

1. 用户在终端执行 `npx 0xmux` 或 `brew install 0xmux && 0xmux`
2. 服务启动并绑定到默认端口 1234
3. 终端输出服务地址：`0xMux running at http://localhost:1234`
4. 用户在浏览器打开该地址，看到极客风命令行风格的管理面板
5. 面板直接展示 tmux session 列表

**预期结果**: 从安装到看到面板，整个流程不超过 2 分钟

### 场景 1b：首次启动（缺少依赖）

**角色**: 开发者
**前置条件**: 用户本机未安装 tmux 或其他必要依赖
**流程**:

1. 用户执行 `npx 0xmux`，服务正常启动（不因缺少 tmux 而阻止启动）
2. 用户在浏览器打开 `http://localhost:1234`
3. 界面展示"环境设置"引导页，以命令行终端风格逐项列出依赖检测结果：
   - 绿色对勾 = 已安装 + 版本号
   - 红色叉号 = 未安装，旁边显示"安装"按钮
   - 黄色感叹号 = 可选依赖未安装（如 Claude Code），旁边显示"安装"按钮或"跳过"
4. 用户点击"安装"按钮，系统在后台执行安装命令（如 `brew install tmux`）
5. 安装进度和日志实时展示在界面上（终端输出风格）
6. 安装完成后，对应依赖项状态自动刷新为绿色对勾
7. 所有必要依赖安装完成后，界面显示"重启服务"按钮
8. 用户点击"重启服务"，后端进程重启，页面自动重连并进入主面板

**预期结果**: 用户无需离开浏览器即可完成所有依赖安装，安装后重启服务自动进入正常使用状态

### 场景 2：日常开发使用

**角色**: 开发者
**前置条件**: 0xMux 已安装，tmux 正在运行多个会话
**流程**:

1. 用户执行 `0xmux` 启动服务
2. 浏览器自动展示所有 tmux session 的实时状态
3. 用户可以创建、关闭、重命名 session
4. 用户可以查看每个 session 的 attached/detached 状态
5. 手机端通过同一局域网地址也能访问和操作

**预期结果**: 所有 session 操作可在浏览器中完成，状态实时同步

### 场景 3：项目开发者贡献代码

**角色**: 开源贡献者
**前置条件**: 已 clone 项目仓库
**流程**:

1. 执行一条命令同时启动前端和后端开发服务器
2. 前端热更新，后端自动重编译
3. 前端通过代理自动转发 API 请求到后端
4. 修改代码后无需手动重启

**预期结果**: 单条命令即可进入开发模式，前后端自动协调

## 功能需求

### FR-1：Monorepo 项目结构

项目采用统一的 monorepo 结构组织前后端代码：

- 根目录包含统一的开发脚本和构建配置
- `web/` 目录存放 React 前端代码
- `server/` 目录存放 Rust 后端代码
- `docs/` 目录存放项目文档
- 根目录的 `package.json` 作为 monorepo 的编排入口，定义统一的 `dev`、`build`、`start` 命令

**验收标准**:

- 根目录存在 `package.json`，包含 `dev`、`build`、`start` 脚本
- `dev` 命令同时启动前端开发服务器和后端开发服务器
- 前端和后端的依赖管理相互独立

### FR-2：前后端通信机制

前端通过两种方式与后端通信：

- **REST API**：用于 Session 的 CRUD 操作（列表、创建、删除、重命名）、系统配置查询、依赖检测与安装触发
- **WebSocket**：用于终端实时数据流传输（后续 xterm.js 集成）、session 状态实时推送、依赖安装日志实时推送

通信约定：
- REST API 路径统一前缀 `/api/`
- WebSocket 端点为 `/ws`
- 开发模式下前端通过 Vite proxy 转发请求到后端
- 生产模式下 Rust 后端直接提供前端静态文件服务

**验收标准**:

- 前端可通过 REST API 获取 session 列表、创建/删除/重命名 session
- WebSocket 连接可建立并传输数据
- 开发模式和生产模式下通信均正常工作

### FR-3：统一开发命令

提供一条命令同时启动前端和后端：

- `npm run dev`（或 `bun dev`）在根目录执行时，并行启动：
  - 前端 Vite 开发服务器（端口 3000）
  - 后端 Rust/Axum 服务器（端口 3001，使用 cargo-watch 实现热重载）
- 两个进程的日志同时输出到终端，通过颜色或前缀区分
- 任一进程退出时自动终止另一个

**验收标准**:

- 根目录执行 `npm run dev` 后，前端和后端均成功启动
- 修改前端代码触发热更新
- 修改后端代码触发自动重编译
- Ctrl+C 一次性终止所有进程

### FR-4：构建与打包流程

构建流程分为前端和后端两部分，最终合并为单一可执行文件：

**前端构建**：
- 使用 Vite 将 React 应用构建为静态文件（HTML/CSS/JS）
- 输出到 `web/dist/` 目录

**后端构建**：
- Rust 编译时将 `web/dist/` 目录的前端静态文件嵌入二进制文件（使用 `rust-embed` 或 `include_dir`）
- 最终产出单个可执行文件，内含前端和后端

**npm 包构建**：
- npm 包作为"启动器"，包含预编译的平台对应二进制文件
- 用户执行 `npx 0xmux` 时，启动器下载或使用内置的对应平台二进制
- 支持平台：macOS (arm64, x86_64)、Linux (x86_64)

**验收标准**:

- `npm run build` 完成前端和后端的完整构建
- 构建产出的单个二进制文件可独立运行
- 二进制文件启动后可通过浏览器访问完整的 Web 界面

### FR-5：多渠道分发

支持三种安装方式：

1. **npx 0xmux**: 无需全局安装，即用即走
2. **npm install -g 0xmux**: 全局安装，长期使用
3. **brew install 0xmux**: macOS 用户的原生体验

安装后执行 `0xmux` 即在本机启动服务，默认绑定 `localhost:1234`。

**验收标准**:

- 通过 `npx 0xmux` 可成功启动服务
- 通过 `npm install -g 0xmux && 0xmux` 可成功启动服务
- Homebrew formula 存在且可安装
- 启动后在终端显示服务地址
- 浏览器访问服务地址可看到管理面板

### FR-6：环境依赖检测与 UI 引导安装

服务启动时不因缺少 tmux 等依赖而阻止启动。环境检测和依赖安装均通过 Web UI 完成：

**启动行为**：
- 0xMux 服务本身始终正常启动（不依赖 tmux 才能运行）
- 启动后通过 `/api/system/deps` 接口向前端报告依赖检测结果

**Web UI 引导页**：
- 首次打开或检测到缺失依赖时，展示"环境设置"引导页
- 引导页采用终端输出风格，逐行显示检测结果：
  - `[✓] tmux 3.4` — 已安装，绿色
  - `[✗] tmux — not found` — 未安装，红色，旁边有"Install"按钮
  - `[!] claude-code — optional, not found` — 可选未安装，黄色，有"Install"和"Skip"按钮

**依赖安装流程**：
- 用户点击"Install"按钮后，后端执行对应的安装命令（macOS 使用 `brew install`，Linux 使用系统包管理器）
- 安装日志通过 WebSocket 实时推送到前端，以终端滚动输出风格展示
- 安装完成后自动重新检测该依赖并刷新状态显示
- 安装失败时展示错误日志并提供手动安装命令供用户复制

**服务重启**：
- 所有必要依赖安装完成后，界面显示"Restart 0xMux"按钮
- 点击后后端进程优雅重启，前端自动等待重连（显示 reconnecting 动画）
- 重连成功后自动跳转到主面板

**检测的依赖项**：

| 依赖        | 类型 | 最低版本 | 安装方式 (macOS)        | 安装方式 (Linux)           |
| ----------- | ---- | -------- | ----------------------- | -------------------------- |
| tmux        | 必需 | 2.6      | `brew install tmux`     | `apt install tmux` / 等    |
| Claude Code | 可选 | -        | `npm i -g @anthropic-ai/claude-code` | 同左 |

**验收标准**:

- 0xMux 服务在 tmux 未安装时也能正常启动并提供 Web 界面
- 打开浏览器后展示环境设置引导页，正确显示每个依赖的检测状态
- 点击"Install"按钮后安装日志实时显示在界面上
- 安装完成后依赖状态自动刷新
- 点击"Restart"后服务重启，前端自动重连并进入主面板
- 所有依赖就绪时，后续启动直接进入主面板（跳过引导页）

### FR-7：极客命令行风格 UI

Web 界面遵循极客/暗色/终端风格设计规范：

- **配色**: 纯黑背景 (#0a0a0a)，霓虹绿强调色 (#00ff41)，灰色文字层级
- **字体**: JetBrains Mono 等宽字体，纯命令行感觉
- **布局**: 高信息密度，卡片式 session 展示，PC 三列 / 手机单列响应式
- **动效**: 最小化，仅状态变化时有微弱霓虹呼吸灯效果
- **交互**: 操作路径短，主要操作不超过 2 次点击
- **终端启动画面**: 服务启动时在终端输出 ASCII Art logo

**验收标准**:

- 界面使用暗色背景和霓虹绿强调色
- 文字使用等宽字体
- PC 端展示三列网格，手机端单列堆叠
- Session 状态通过颜色直观区分（绿色 = attached，灰色 = detached）

### FR-8：服务配置

- 默认端口 1234，支持通过命令行参数或环境变量自定义
- `0xmux --port 8080` 或 `PORT=8080 0xmux`
- 绑定地址默认 `127.0.0.1`（仅本机访问），可通过 `--host 0.0.0.0` 开放局域网访问

**验收标准**:

- 默认启动在 1234 端口
- `--port` 参数可修改端口
- `--host` 参数可修改绑定地址

## 关键实体

| 实体           | 说明                                                       |
| -------------- | ---------------------------------------------------------- |
| Session        | tmux 会话，包含名称、窗口数、创建时间、attached 状态       |
| Window         | tmux 窗口，属于某个 Session                                |
| Pane           | tmux 窗格，属于某个 Window                                 |
| 依赖项         | 名称、类型（必需/可选）、检测状态、版本号、安装命令        |
| 安装任务       | 依赖项安装的执行记录，包含状态（进行中/成功/失败）和日志   |
| 服务配置       | 端口、绑定地址、日志级别等运行时参数                       |

## 假设

1. 用户本机已安装 Node.js >= 18 或 Homebrew（用于安装 0xMux）
2. tmux 可能未安装，服务启动不依赖 tmux，依赖安装通过 Web UI 引导完成
3. Claude Code 为可选依赖，不影响核心功能
4. 用户使用 macOS 或 Linux 系统（暂不支持 Windows）
5. npm 包通过预编译二进制 + 平台检测方式分发，而非要求用户安装 Rust 工具链
6. 开发模式下使用 `concurrently` 或类似工具同时运行前后端
7. 生产模式下 Rust 二进制内嵌前端静态文件，无需单独部署前端
8. 默认端口选择 1234 以避免与常见开发端口（3000、8080 等）冲突
