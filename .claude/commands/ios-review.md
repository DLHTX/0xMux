# iOS Code Review 命令

对 iOS/Swift 代码进行质量审核，检测冗余、重复、明显的 bug 和架构问题。

## 使用方式

```
/ios-review <path>          # 审核指定路径
/ios-review .               # 审核当前目录
/ios-review --quick <path>  # 快速扫描，只报告关键问题
```

## 执行步骤

1. **确定审核范围**
   - 解析用户提供的路径
   - 列出所有 `.swift` 文件
   - 统计文件数量和代码行数

2. **冗余检测**
   - 搜索相似的代码模式
   - 识别重复的逻辑块
   - 检查是否需要提取常量

3. **Bug 检测**
   - 扫描 force unwrap ("!")
   - 检查闭包捕获列表 ([weak self])
   - 验证主线程 UI 更新
   - 查找逻辑错误

4. **架构审计**
   - 统计每个功能模块的文件数
   - 分析 import 语句
   - 检查文件大小（>500 行标记）
   - 验证层级边界

5. **生成报告**
   - 按严重程度排序
   - 提供可操作的建议
   - 包含修复代码示例

## 审核维度

### 1. 冗余与重复 (Redundancy)

检查项：
- 重复代码块（>10 行相似）
- 功能相似的函数
- 未提取的字符串常量
- 重复的错误处理模式

严重程度：
- 🔴 **高**: >30 行重复，3+ 位置
- 🟡 **中**: 15-30 行重复，2 位置
- 🔵 **低**: <15 行或轻微重复

### 2. 明显 Bug (Bugs)

检查项：
- Force unwrap 没有 nil 检查
- 闭包中缺少 [weak self] 导致循环引用
- delegate 未声明为 weak
- 观察者未在 deinit 中移除
- 主线程违规（后台线程更新 UI）
- 逻辑错误（不可达代码、条件反转）

严重程度：
- 🔴 **致命**: 会导致崩溃或数据损坏
- 🟡 **警告**: 特定条件下可能出问题
- 🔵 **提示**: 潜在问题，值得关注

### 3. 架构问题 (Architecture)

#### 过度拆分 (Over-Fragmentation)
- 单方法的类/结构体
- 只有一个实现的协议（且无测试 mock）
- 小功能拆成太多文件
- 深层嵌套（>4 层）
- <50 行的文件应合并

#### 高耦合 (High Coupling)
- 模块间循环引用
- 未使用依赖注入
- 巨型 ViewController（>500 行）
- 上帝对象（知道太多）
- 层级越界（UI 直接访问数据库等）

```markdown

- **关键问题**: X
- **警告**: X
- **提示**: X

### 1. [类别] 问题标题
**位置**: FileName.swift:LINE
**描述**: 简要说明
**问题代码**:
​```swift

​```

​```swift
// 修复后的代码
​```

## 警告 (建议修复)
...

## 提示 (可考虑)

- **拆分度**: 低/中/高
- **整体评估**: 简要总结

## 注意事项

1. 实用主义 - 不是每个重复都需要提取
2. 考虑项目阶段 - 早期可以容忍一些技术债
3. 优先处理影响可靠性的问题
4. 始终提供具体、可操作的建议
5. 非简单问题要包含代码示例
